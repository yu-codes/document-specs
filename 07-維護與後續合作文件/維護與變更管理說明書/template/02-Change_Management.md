# 肆、變更申請流程

## 4.1 變更類型定義

| 變更類型 | 定義 | 核准層級 | 處理時限 | 範例 |
|---------|------|---------|---------|------|
| 標準變更 | 低風險、已有流程之例行變更 | 維護窗口自行執行 | [填寫] 個工作天 | 套件更新、設定微調 |
| 一般變更 | 需評估影響的功能變更 | 雙方窗口核准 | [填寫] 個工作天 | 功能修改、UI 調整 |
| 緊急變更 | 影響服務之緊急修復 | 事後補核准 | 立即處理 | 嚴重 Bug 修復 |
| 重大變更 | 高風險或大範圍變更 | 雙方主管核准 | [填寫] 個工作天 | 架構變更、資料庫修改 |

## 4.2 變更申請格式

| 欄位 | 說明 |
|------|------|
| **變更編號** | CR-[YYYY]-[序號] |
| **申請日期** | YYYY-MM-DD |
| **申請人** | [填寫] |
| **變更類型** | [標準/一般/緊急/重大] |
| **優先級** | [高/中/低] |
| **變更標題** | [簡述變更內容] |
| **變更描述** | [詳細說明變更需求] |
| **變更原因** | [為何需要此變更] |
| **影響範圍** | [受影響之模組/功能/使用者] |
| **時程影響** | [對時程之影響評估] |
| **費用影響** | [是否產生額外費用] |
| **風險評估** | [變更可能之風險] |
| **回滾計畫** | [失敗時如何回復] |

## 4.3 變更評估流程

```
收到變更申請
    ↓
可行性評估
    ↓
影響分析（時程/費用/風險）
    ↓
提出評估報告
    ↓
客戶確認
    ↓
核准 → 排入工作計畫
  或
駁回 → 通知申請人
```

### 評估內容

| 評估項目 | 說明 |
|---------|------|
| 技術可行性 | 技術上是否可實現 |
| 影響範圍 | 對現有功能的影響 |
| 工時估計 | 開發、測試所需時間 |
| 風險評估 | 可能的風險與緩解措施 |
| 費用估計 | 是否超出維護範圍需額外計費 |

## 4.4 變更審核與核准

| 變更類型 | 核准人 | 核准方式 | 備註 |
|---------|--------|---------|------|
| 標準變更 | 乙方維護人員 | 自行判斷 | 事後記錄 |
| 一般變更 | 雙方專案窗口 | Email 確認 | 需書面紀錄 |
| 緊急變更 | 乙方技術主管 | 電話/即時 | 事後補簽 |
| 重大變更 | 雙方主管 | 正式簽核 | 會議討論 |

## 4.5 變更實施流程

1. **規劃**：確認實施時間、影響範圍、通知對象
2. **準備**：建立備份、準備回滾方案
3. **實施**：依計畫執行變更
4. **驗證**：測試變更結果
5. **確認**：通知客戶確認
6. **結案**：更新文件、關閉變更單

# 伍、版本控管

## 5.1 版本編號規則

```
主版本.次版本.修訂版本
例: 1.2.3
```

| 版本類別 | 變更時機 | 範例 |
|---------|---------|------|
| 主版本 (Major) | 重大架構變更、不相容修改 | 1.0.0 → 2.0.0 |
| 次版本 (Minor) | 功能新增（向後相容） | 1.0.0 → 1.1.0 |
| 修訂版本 (Patch) | Bug 修復、安全更新 | 1.0.0 → 1.0.1 |

## 5.2 發佈流程

| 步驟 | 動作 | 環境 | 負責人 | 確認方式 |
|------|------|------|--------|---------|
| 1 | 開發完成 | Development | 開發人員 | Code Review 通過 |
| 2 | 整合測試 | Staging | 測試人員 | 測試報告通過 |
| 3 | 客戶驗收 | Staging | 客戶 | 書面確認 |
| 4 | 正式部署 | Production | 維運人員 | 部署檢核表 |
| 5 | 上線驗證 | Production | QA | 冒煙測試通過 |
